      1. Introduction

      1.1 Purpose

      In this data-driven technological era, there is an abundance of data readily accessible to all with the help of search engines. But users, especially researchers and such may face hurdle to get specific results for their need with the overwhelming amount of data available. This is where Search Cluster, a cluster-based search application comes in to play as its main purpose is to let its users to search in their custom-made clusters of links with the type of files the users are looking for. This document serves to provide a detailed description of the requirements for the software Re-search, the details of its features and functionalities, and the constrains it must run under. This document is made for version 1.0 of the software.

      1.2 Document Conventions

      Standard document conventions were followed while preparing this document with bold fonts for main topic headers, and had larger font size than the sub topics, body texts etc. Sub topics were divided into different sections under main topics with the main topic number as the prefix. Priority ratings in the later section range from 1 to 9, 9 being the most important and 1 being the least. Special names and technical terminology are italicized, which have been explained later in the document in the glossary section (Appendix A).

      1.3 Intended Audience and Reading Suggestions

      The intended audiences of this document would be the users, mainly researchers, academics or students who intend to use the software and the document would be a very helpful guide for them to start on as the SRS covers all of the aspects of the software Re-search in six different sections in an organized and sequential manner with little to no technical jargons. The readers are suggested to go through all of the sections to get a good grasp about the ins and outs of the software. The first section gives an introduction of the product, the second section deals with the descriptions about its functions, classes and such. The third section illustrates all of the external interface requirements, fourth section talks about the software's features. Fifth and Sixth sections deal with various other requirements.

      1.4 Product Scope

      Re-search is designed to make searching very specific for its users. Users will be able to set their own clusters of links and will be able to define search methods based on textual file types such as: pdf, txt, doc etc. Users then will be able to search for keywords within the clusters to get intended results. The software serves to make researching easier for its users as other conventional search engines can often provide overwhelming number of results or data that can make the research very cumbersome.

      1.5 References

      • Django Framework
      • Apache Lucene
      • Scrapy Web Crawler

      2. Overall Description

      2.1 Product Perspective

      Re-search is an application that enables user to search within website of their selection. Typical Google search comb through the entire internet. This can be useful for most cases but for certain case like research, data manipulation restricted search can be useful.

      2.2 Product Functions

      • Admin can create or add new search method.
      • User can create their profile. They can create a search cluster containing one or multiple website and they also choose the search method such as text, pdf, ppt etc. Once cluster been created he will get a confirmation message when search function is applicable.
      • User can search for a keyword from his own cluster and view those result also visit the url that contains his search keywords. He can export his search result page into a xml file.

      2.3 User Classes and Characteristics

      Admin: Add new search method as plugins. They will have control over crawling methods.

      Client: They will have acces to their account. They can create search cluster and do search from there. Client will have limited class privilege. They can only use service but can not modify them.

      3. External Interface Requirements

      3.1 User Interfaces

      The user interface will provide users with a way of interacting with the Re-search application. It will be the first thing that the user will notice. The Re-search user interface will be designed in such a way so that the users can quickly figure out how to use the application. The Re-search application contains following user interfaces:

      • Login page will accept user id and password to authenticate registered users. It will also provide options like Registration and Sign up using Google for new users.

      • After successful registration or login, the home page will appear with a menu; where the user will be able to find the options 'Create a new cluster' and 'View existing clusters' to see the clusters the user will create. The users will be able select options from the menu that will redirect the specific page or screen. If a user faces any difficulties to understand the functionality, then he/she can seek for help by clicking Help button.

      • In each Cluster page, the URLs user has added will be shown in a table. There will be a button to add URL, a search box that can accommodate 50 characters so that the users can write longer queries being able to see the see the search text and check the typos, and a dropdown menu to select to the type of output user wants.

      • Search results will be shown in a different page or screen in a paginated view for web application. It also has an Export button. User can click the button to export search results. The mobile application user interface is a bit different. It has the options to sign in and search. Search results in the mobile application are shown in list view. It will also have the Export option to export search result.

      3.2 Hardware Interfaces

      Since the main purpose of the application is searching, an active internet connection is required to communicate with the server, to fetch search results and to make API calls. There is no such special requirement of hardware for the Search Cluster application as crawl data from websites will be stored in the server. However, a clear and working microphone is required for the mobile application to perform search via voice command.

      3.3 Software Interfaces

      The Re-search web application will run on any operating system that is capable with modern web browsers. Both the android and web application will communicate with database via internet. Django is used to interact with the centralized server, while Djnago default database SQLite is used to store data. The android application will communicate with the backend via REST API.

      Scrapy is used for web scraping. It is an open-source web-crawling framework which is free. Scrapy provides all the tools and efficient ways to extract data from websites, process them and store them in a preferred format and structure. Apache Lucene will be used for indexing and searching data in a more efficient way. It is an open source search engine library that provides powerful features for indexing and searching.

      3.4 Communication Interfaces

      The Re-search application will use standard HTTP protocol communication. Sign up authentication will be done via email. After creating a Search cluster the user have to wait until the system crawls data from user’s preferred websites. The system will notify the user via email once it becomes ready to perform search operation. All these will be done through HTTP requests.

      4. System Features

      The users will use the Re-search software, admin will control the search method and fianlly system will do some automated task.

      4.1 Signup

      Use Case Name:   Signup
      Precondition:   None
      Actor(s):   Client
      Goal:   To create a user account.

      Overview:   The client will create their account to use this application. They need to provide a valid email and password to create their account. Clients can sign up using their Google accounts also. The user information is saved in the database.
      Priority:   High

      Actor actions System responses
      1. The user enters email and password and then clicks on "Signup" button 2. Authenticates the user information.
      3. Stores user information in the database.
      4. Displays account created successfully.

      Alternative courses:

      Step 1-2: User can also use Google signup method instead of entering email and password. The system will automatically create user account based on Google authentication.

      4.2 Login

      Use Case Name:   Login
      Precondition:   User account must be created.
      Actor(s):   Admin, client
      Goal:   The user will be able to log into the system.

      Overview:   Actor enters valid email and password to log in. Then the actor is redirected to the home page or screen based to the type of actor.
      Priority:   High

      Actor actions System responses
      1. The user enters their email and password and then click on "Login" button 2. Identifies the user.
      3. Redirects the user to the respective home page.

      Alternative courses:

      Step 1: User can use google login method instead of entering email and password. The system automatically verifies the user account and accepts login request.
      Step 2-3: User enters invalid email or password. The system fails to identify the user and it will ask user to enter information again in that case.

      4.3 Create search cluster

      Use Case Name:   Create search cluster.
      Precondition:   Client account must be created and Client must be logged in.
      Actor(s):   Client
      Goal:   To create a search cluster.

      Overview:   The Client will create search cluster. Each cluster can have one or multiple urls. Clients will decide the crawling method, wheather it will be pdf, text or any other system provided crawling method.They can name their cluster according to their choice.
      Priority:   High

      Actor actions System responses
      1. The Client will set a name for the Cluster and click on “Create cluster” button. 2. Stores cluster details in database.
      3. Displays Cluster created successfully.

      Alternative courses:

      Step 2-3: The client might cancel the cluster creation process.

      4.4 Modify search cluster

      Use Case Name:   Modify search cluster.
      Precondition:   Search cluster must be created.
      Actor(s):   Client
      Goal:   Client can modify own created cluster and add new urls.

      Overview:   The client will be able add new urls along with search depth to an existing search cluster.
      Priority:   Medium

      Actor actions System responses
      1. The client selects an existing cluster. 2. Displays cluster details along with add url option.
      3. The clinet selects add url option. 4. Displays text box along with set depth option.
      5. The client adds a new url with search depth. 6. Stores url and search depth for the cluster in the database.

      Alternative courses:

      Step 3-6: The client might not want to modify cluster. The process is terminated.

      4.5 Search for a keyword from own cluster

      Use Case Name:   Search for a keyword from own cluster.
      Precondition:   Search cluster must be created.
      Actor(s):   Client
      Goal:   Client search for a keyword from their cluster.

      Overview:   The client will be able to do search operation. The Client selects own created cluster and search a keyword. They system shows the search result in a paginated view.
      Priority:   High

      Actor actions System responses
      1. The client enters a keyword and click of the “Search” button. 2. Searches for that keyword from specified search cluster’s urls.
      3. Displays matching search results in a paginated view.

      Alternative courses:

      Step 3-4: Search Cluster is not yet ready to perform search operation. System displays a warning message.
      Step 4: Could not find any matching result for the given text. System displays no such result found.

      4.6 Visit webpage

      Use Case Name:   Visit webpage
      Precondition:   Client must do a search operation.
      Actor(s):   Client
      Goal:   To visit a webpage that is containing the matching search result.

      Overview:   The client will be able to see the list of website that contains his keyword. Client can select one of those results that redirect the client to the webpage that is containing the search result.
      Priority:   High

      Actor actions System responses
      1. Client clicks on the link to visit the website. 2. Redirects to the destination page that is containing the matching result.

      Alternative courses:

      Step 1-2: In case no matching result found, the client cannot visit any webpage.

      4.7 Export search result

      Use Case Name:   Export search result.
      Precondition:   Client must do search operation that should find some matching results.
      Actor(s):   Client
      Goal:   To save the search results in local machine.

      Overview:   Client can save all the matching result of his search result into a xml file for later usage.
      Priority:   Medium

      Actor actions System responses
      1. Client clicks on the “Export” button to save the search result as xml file. 2. System converts the search results into xml file and being downloaded.

      Alternative courses:

      Step 1-2: In case of no matching result this operation is not applicable.

      4.8 Logout

      Precondition:   Users must be logged in to the system
      Actor(s):   Client, Admin
      Goal:   To end the user's session

      Overview:   User click on the logout button and this will end his session on the website.
      Priority:   Low

      1. User click on the logout button 2. The system will end the user session and redirect him to the login page.

      Alternative courses:

      Not applicable.

      4.9 Add new crawling method as plugin

      Use Case Name:   Add new crawling method as plugin
      Precondition:   Admin must be logged in to the system.
      Actor(s):   Admin
      Goal:   To add new crawling method as plugin.

      Overview:   Admin choose to add new search method for webcrawling. So he add a new type of crawling method as plugin and the system let him that plugin added successfully.
      Priority:   Medium

      Actor actions

      4.10 Index content of url form search cluster

      Use Case Name:   Index content of url form search cluster
      Precondition:   User must create search cluster and modify own search cluster. operation.
      Actor(s):   System
      Goal:   To index the url content for efficient and fast search.

      Overview:   System will crawl through user given website and fetch the content. After fecthing system will automatically index those content and store them in a server.
      Priority:   High

      Precondition:   System must perform index content of url(s) from search clusters.
      Actor(s):   System
      Goal:   To inform the client that the search cluseter are ready for searching.

      Overview:   System do the indexing url content in batch. After completing the operation the system will automitically inform the user that they can now use the search cluster for search operation.
      Priority:   Low

      Actor actions System responses
      1. Client create and modify the search cluster. 2. System index the url content in batch and after finishing the task an automated message will be sent to the client that search cluster is ready now.

      Alternative courses:

      Step 1-2: User add invalid url and system can not do index operation so invalid message will be sent.

      5. Other Nonfunctional Requirements

      5.1 Performance Requirements

      A cluster defined by a user may take up to 24 hours before it becomes searchable. Crawling also depends on the performance of the crawling framework that is used, in this context, scrapy. For example, the Scrapy Benchmarking documentation states that scrapy may be able to crawl about 3000 pages per minute. Performance may vary depend on how much workload the crawler may take on and how well it’s written.

      5.2 Safety Requirements

      Users of Re-search are advised to only provide for links of websites that allow crawling their webpages. Users are expected to be responsible individuals whom respect and follow the crawling policies of the links that they provide.

      5.3 Security Requirements

      The software requires an email, a password for each user to register and also login to use the software. There is also another option for users to sign up/authenticate with their Gmails. Crawling restrictions provided by particular websites to serve crawling requests may also be an issue in certain contexts. As Re-search is made by following the Fair Usage Policies (FUP), the software cannot display the contents that it crawls due to the issues of copyrights.

      5.4 Software Quality Attributes

      The software quality attributes of Re-search are:

      • Availability:: The software will be available to use at all times for its users except for scheduled maintenance of servers as mentioned in the Business Rules section.
      • Correctness: The software will provide accurate results based on the links the users provide in clusters and the keywords they will use when they need to search.
      • Reusability: The users will be able to reuse and search many times on the clusters that they create.

      5.5 Business Rules

      The software will be unavailable to use at times for maintenance of the servers for 2 hours every month. As of the release of Re-search’s version 1.0, the software is intended to be free for its users and does not contain any added premium features or plugins.

      Appendix A: Glossary

      • Data Crawling Ethics: Repeated requests of the web pages can slow down the webserver. Sometimes aggressive scraping may lead to functional issues for the website. Besides, websites may contain materials with copyright claims. Making copies of those copyright materials is considered as an illegal activity. So, web crawling should be done at a disruptive rate by maintaining website's terms and conditions so that it does not slow down the website and hamper it's regular activities.
      • Fair usage policy: Web scraping and crawling is not illegal unless it is done in an unethical way. Scraping private informations from website and exposing it publicly falls under illegal activity. Scrapped data should not be used for commercial purposes.
      • API: Application Programming Interface (API) provides an interface between two pieces of software to communicate. API allows an application to access data and interact with external components of the software.
      • Framework: Framework is used as a foundation for application developments. It allows application developers to develop new features in a well-structured manner. Framework provides generic functionality that can be changed to use for specific purposes.
